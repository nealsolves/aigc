# v0.3.2 Design Spec — Split Enforcement

Date: 2026-04-04
Status: Accepted
Baseline: `v0.3.1`

---

## 1. Release Context and Baseline

AIGC `v0.3.1` ships a unified enforcement model:

- the host provides a complete invocation including `output`
- `enforce_invocation()` evaluates the full gate sequence after model execution
- `@governed` executes the wrapped function first and enforces afterward
- audit artifacts use schema version `1.2`

`v0.3.2` is an additive design release that introduces split enforcement without
breaking the released `v0.3.1` public contract.

The core intent is straightforward: authorization and other pre-call checks that
can be decided before model execution should happen before token spend, while
output-dependent checks remain after the call.

## 2. Scope and Non-Goals

`v0.3.2` adds:

- `enforce_pre_call()` and `enforce_post_call()`
- async parity for split entry points
- `PreCallResult` as the Phase A to Phase B handoff object
- additive audit artifact schema `v1.3`
- split-mode support in `@governed`
- instance-scoped split methods on `AIGC`

`v0.3.2` does not add:

- provenance or lineage features
- queryable history beyond the existing artifact chain
- new built-in governance gates
- a default flip to `pre_call_enforcement=True`
- a two-artifact split model
- `v0.4.0` architecture work

## 3. Architectural Invariants Governing the Change

The split design is constrained by these invariants:

1. Gate ordering remains fixed.
2. Enforcement remains fail-closed.
3. Exactly one audit artifact is emitted per invocation attempt.
4. Unified mode remains backward-compatible.
5. Policy evaluation in Phase B must use the Phase A effective policy and must
   not reload policy from disk.
6. Split mode must preserve the same failure taxonomy unless an additive change
   is strictly necessary.
7. Audit schema evolution is additive only.

These invariants are more important than implementation convenience. Any split
implementation that weakens them is out of scope.

## 4. Current Unified Enforcement Model

In `v0.3.1`, a valid invocation requires:

- `policy_file`
- `model_provider`
- `model_identifier`
- `role`
- `input`
- `output`
- `context`

Unified enforcement runs one ordered pipeline:

1. load policy
2. custom gates at `pre_authorization`
3. guard evaluation and condition resolution
4. role validation
5. precondition validation
6. tool constraint validation
7. custom gates at `post_authorization`
8. custom gates at `pre_output`
9. output schema validation
10. postcondition validation
11. custom gates at `post_output`
12. risk scoring
13. audit artifact generation and emission

This model is retained as a supported public mode in `v0.3.2`.

## 5. v0.3.2 Split Enforcement Model

`v0.3.2` introduces two explicit phases around the model call.

### 5.1 Phase A: Pre-Call Enforcement

Phase A accepts an invocation without `output` and runs only gates that do not
require model output:

1. load policy
2. custom gates at `pre_authorization`
3. guard evaluation and condition resolution
4. role validation
5. precondition validation
6. tool constraint validation
7. custom gates at `post_authorization`

If Phase A passes, the SDK returns a `PreCallResult`.

If Phase A fails, the SDK emits one final FAIL artifact and raises a typed
exception. The wrapped model call must not execute.

### 5.2 Phase B: Post-Call Enforcement

Phase B consumes a `PreCallResult` plus `output` and runs the remaining gates:

1. custom gates at `pre_output`
2. output schema validation
3. postcondition validation
4. custom gates at `post_output`
5. risk scoring
6. audit artifact generation and emission

Phase B returns the final artifact on PASS and raises with an attached FAIL
artifact on FAIL.

## 6. Gate Ordering Preservation Proof

The split model does not reorder gates. It only moves the model call boundary.

Unified relative order:

- authorization-side work finishes before output-side work starts
- output-side work finishes before risk scoring and final artifact generation

Split relative order:

- Phase A executes the same authorization-side sequence in the same order
- the model call occurs after `post_authorization`
- Phase B starts at `pre_output` and preserves the rest of the order unchanged

Therefore:

- no gate crosses another gate
- only the host call boundary moves
- the proof obligation from `v0.3.1` remains satisfied

## 7. Public API Surface

`v0.3.2` adds these public entry points:

```python
from collections.abc import Mapping
from typing import Any


def enforce_pre_call(invocation: Mapping[str, Any]) -> PreCallResult: ...
async def enforce_pre_call_async(invocation: Mapping[str, Any]) -> PreCallResult: ...

def enforce_post_call(
    pre_call_result: PreCallResult,
    output: dict[str, Any],
) -> dict[str, Any]: ...

async def enforce_post_call_async(
    pre_call_result: PreCallResult,
    output: dict[str, Any],
) -> dict[str, Any]: ...
```

The `AIGC` class adds instance-scoped equivalents with the same semantics.

Existing public APIs remain supported:

- `enforce_invocation()`
- `enforce_invocation_async()`
- `@governed()`

Unified mode remains the default compatibility path.

## 8. Invocation Validation Factoring

### 8.1 `enforce_pre_call` Invocation Shape

`enforce_pre_call()` accepts exactly the unified invocation shape minus `output`.

Required keys:

- `policy_file: str`
- `model_provider: str`
- `model_identifier: str`
- `role: str`
- `input: dict[str, Any]`
- `context: dict[str, Any]`

`output` is not required and must not be used as a placeholder requirement.

### 8.2 Validation Factoring Strategy

Internal invocation validation is factored into three layers:

1. `_validate_invocation_core(invocation)`
   - validates mapping type
   - validates the four required string identity keys
   - validates `input` is a mapping-like dict payload
   - validates `context` is a mapping-like dict payload
   - validates JSON-serializability of common fields
2. `_validate_invocation(invocation)`
   - calls `_validate_invocation_core(invocation)`
   - validates `output` exists and is a dict
   - preserves current unified behavior
3. `_validate_pre_call_invocation(invocation)`
   - calls `_validate_invocation_core(invocation)`
   - does not require `output`

This preserves `v0.3.1` unified behavior while enabling split mode safely.

## 9. `PreCallResult` Data Contract

`PreCallResult` is a module-level dataclass with frozen slots.

```python
@dataclass(frozen=True, slots=True)
class PreCallResult:
    effective_policy: Mapping[str, Any]
    resolved_guards: tuple[dict[str, Any], ...]
    resolved_conditions: Mapping[str, Any]
    phase_a_metadata: Mapping[str, Any]
    invocation_snapshot: Mapping[str, Any]
    policy_file: str
    model_provider: str
    model_identifier: str
    role: str
    _consumed: bool = field(init=False, default=False, repr=False, compare=False)
```

Field intent:

- `effective_policy`: resolved policy after guard expansion
- `resolved_guards`: immutable record of matched guard expansions
- `resolved_conditions`: resolved condition values used in Phase A
- `phase_a_metadata`: ordered Phase A evidence and timestamps needed by Phase B
- `invocation_snapshot`: canonical pre-call invocation state
- `policy_file`, `model_provider`, `model_identifier`, `role`: identity fields
- `_consumed`: internal lifecycle bit, not public API

### 9.1 Exact `invocation_snapshot` Fields

`invocation_snapshot` captures exactly:

- `policy_file`
- `model_provider`
- `model_identifier`
- `role`
- `input`
- `context`

It does not include `output`.

## 10. `PreCallResult` Lifecycle and One-Time Consumption Model

`PreCallResult` is logically immutable with one internal lifecycle bit.

### 10.1 Why `frozen=True, slots=True`

This shape is required because it:

- prevents accidental host mutation of governance-relevant state
- avoids dynamic attribute injection
- keeps the handoff object compact and predictable
- makes misuse easier to detect

### 10.2 `_consumed` Contract

`_consumed` is declared as:

```python
_consumed: bool = field(init=False, default=False, repr=False, compare=False)
```

Rules:

- it is initialized to `False`
- it is not part of the public governance contract
- it is mutated only internally via `object.__setattr__`
- it exists solely to enforce one-time use of a pre-call token

### 10.3 One-Time Consumption

`enforce_post_call()` performs this sequence:

1. validate that the first argument is a `PreCallResult`
2. validate that `output` is a dict
3. read `_consumed`
4. if `_consumed` is `True`, fail closed with `InvocationValidationError`
5. otherwise set `_consumed = True` with `object.__setattr__`
6. continue with Phase B

A `PreCallResult` is therefore valid for one Phase B attempt only.

### 10.4 Concurrency Contract

`PreCallResult` is single-threaded. Concurrent consumption of the same object is
unsupported and documented as undefined behavior.

This is acceptable because:

- `@governed` creates and consumes it within one call scope
- correct manual split usage is one pre-call token per one post-call execution
- adding synchronization for host misuse is disproportionate complexity for
  `v0.3.2`

### 10.5 Pickling Contract

Pickle round-trip preserves both `_consumed` state and provenance.

Consequences:

- an unconsumed token remains unconsumed after round-trip and passes
  `enforce_post_call()` provenance checks
- a consumed token remains consumed after round-trip and is still rejected
- hosts that need a fresh token must call `enforce_pre_call()` again

`_consumed` is a plain `bool` field and is preserved by standard dataclass
pickling.

`_origin` holds an `_EnforcementToken` singleton whose `__reduce__` returns
`(_get_enforcement_token, ())`. On unpickle Python calls
`_get_enforcement_token()` which returns the process-local module singleton,
preserving object identity for the `is` check in `enforce_post_call()`.
`__deepcopy__` returns `self` for the same reason.

These two behaviors together mean a pickle or deepcopy round-trip of an
unconsumed `PreCallResult` produces a token that `enforce_post_call()` will
accept.

### 10.6 Host Responsibility vs SDK Responsibility

The SDK detects structural misuse:

- wrong type passed to `enforce_post_call`
- already-consumed `PreCallResult`
- invalid `output` payload shape

The SDK does not detect logical staleness such as:

- a host holding a pre-call token too long
- a host pairing a token with the wrong external model execution
- policy changes on disk after Phase A

This is a host responsibility in manual split mode. `@governed` avoids this
problem by performing both phases in one wrapper lifecycle.

#### Provenance check threat model

`enforce_post_call()` rejects `PreCallResult` objects that were not produced
by `enforce_pre_call()`. This is **misuse-detection** with HMAC defense-in-depth,
not adversarial in-process security:

- Two layers of checks are performed:
  1. `_origin is not _ENFORCEMENT_TOKEN` — rejects tokens where Phase A never ran.
  2. HMAC-SHA256 over `_frozen_evidence_bytes` with a session key held in a
     module-level closure — rejects tokens that set `_origin` without also
     holding the signing key.
- These layers catch accidental direct construction and raise the bar against
  trivial in-process forgery (importing the sentinel and stamping `_origin`
  no longer suffices; the HMAC must also be computed with the session key).
- They do **not** prevent a determined attacker who extracts the signing key
  via closure introspection (e.g. `_token_sign.__closure__[0].cell_contents`).
  Python offers no language-level protection against this.
- Cross-process or multi-tenant isolation is the host's responsibility. If
  adversarial in-process code is a concern, the host must enforce
  process-level isolation.

## 11. Audit Artifact `v1.3` Design

Schema `v1.3` is additive over `v1.2`.

### 11.1 New Metadata Fields

`metadata` gains these optional fields:

- `enforcement_mode`: `unified` | `split` | `split_pre_call_only`
- `pre_call_gates_evaluated: list[str]`
- `post_call_gates_evaluated: list[str]`
- `pre_call_timestamp: int`
- `post_call_timestamp: int`

No new required top-level fields are added.

### 11.2 Artifact Emission Rules

Exactly one artifact is emitted per invocation attempt.

- Phase A FAIL:
  - emit one final FAIL artifact
  - set `metadata.enforcement_mode = "split_pre_call_only"`
  - do not execute the wrapped model call
- Phase A PASS then Phase B PASS:
  - emit one PASS artifact
  - set `metadata.enforcement_mode = "split"`
- Phase A PASS then Phase B FAIL:
  - emit one FAIL artifact
  - set `metadata.enforcement_mode = "split"`
- Phase A PASS then wrapped function error (Phase B never runs):
  - emit one FAIL artifact
  - set `failure_gate = "wrapped_function_error"`
  - set `metadata.enforcement_mode = "split"`
- Unified mode:
  - emit one artifact
  - set `metadata.enforcement_mode = "unified"`

### 11.3 `output_checksum` on Phase-A-Only FAIL

Phase A fail artifacts use the checksum of `{}` as the `output_checksum`.

This preserves:

- schema validity
- deterministic artifact shape
- compatibility with existing checksum semantics

### 11.4 Failure-Gate Taxonomy

`v0.3.2` introduces one additive `failure_gate` enum value:

- **`wrapped_function_error`** — emitted when the wrapped function raises an
  exception in `@governed(pre_call_enforcement=True)` mode after Phase A PASS
  and before Phase B. Downstream consumers that enumerate `failure_gate`
  values must add it when upgrading to `v0.3.2`.

Structural Phase B misuse (invalid or forged `PreCallResult`) maps to
`invocation_validation`.

## 12. Unified-Mode Backward Compatibility

Unified mode remains the compatibility contract for existing callers.

Unchanged behavior:

- invocation still requires `output`
- `enforce_invocation()` and `enforce_invocation_async()` remain supported
- exception types remain the same
- attached `exc.audit_artifact` behavior remains the same
- the wrapped function in default decorator mode still runs before governance

Internal implementation may route unified mode through split internals, but the
public contract must remain the same.

## 13. `@governed` Behavior and Migration

`@governed` adds:

```python
def governed(..., pre_call_enforcement: bool = False): ...
```

Behavior:

- `False`:
  - preserve existing function-first unified enforcement behavior
- `True`:
  - run `enforce_pre_call()` before executing the wrapped function
  - skip the wrapped function entirely on Phase A FAIL
  - run `enforce_post_call()` after the wrapped function returns a dict output

Migration guidance:

- existing integrations need no change
- callers that want pre-call protection opt in explicitly
- docs and examples must show both modes, with unified mode as the default

## 14. `AIGC` Instance-Scoped Split Methods

The `AIGC` class adds instance-scoped split methods with the same semantics as
module-level functions.

Design rules:

- instance-scoped Phase A uses the existing `PolicyCache`
- instance-scoped Phase B uses `PreCallResult.effective_policy` and must not
  reload policy
- sink behavior and `on_sink_failure` behavior remain unchanged
- sync and async instance paths must produce equivalent stable artifact fields

## 15. Async Parity

Every split public API must have an async equivalent with the same contract.

Required parity points:

- identical validation rules
- identical gate ordering
- identical artifact semantics
- identical failure mapping
- no sync-only metadata fields

The async path must not become a second design. It is the same design with async
execution boundaries.

## 16. Retry, Telemetry, Risk, Signing, and Audit-Chain Interactions

### 16.1 Retry

`with_retry()` remains a unified-mode helper in `v0.3.2`.

Split-mode guidance:

- do not retry Phase A authorization failures
- host-managed retries may wrap the model call plus `enforce_post_call()` for
  transient schema-related cases where that behavior is explicitly desired
- no new split retry helper is required in `v0.3.2`

### 16.2 Telemetry

Span names:

- unified mode: `aigc.enforce_invocation`
- split mode Phase A: `aigc.enforce_pre_call`
- split mode Phase B: `aigc.enforce_post_call`

Required attribute:

- `aigc.enforcement_mode`

Telemetry must remain behaviorally optional. If telemetry is unavailable, the
SDK still enforces normally.

### 16.3 Risk Scoring

Risk scoring remains Phase B only.

There is no Phase A risk computation in `v0.3.2`.

### 16.4 Signing and Audit Chain

Signing and audit-chain behavior remain unchanged.

The artifact payload evolves normally with new optional metadata, but the chain
and signature flows still operate on one final artifact per attempt.

## 17. Pre-Pipeline Failure Handling

Split mode must integrate with `_generate_pre_pipeline_fail_artifact()` for
failures that happen before the main enforcement pipeline starts.

This includes:

- invalid pre-call invocation shape
- invalid `PreCallResult` passed into `enforce_post_call()`
- invalid Phase B `output` payload shape
- policy-load failures that happen before Phase A begins

Design rules:

- pre-pipeline split failures produce schema-valid FAIL artifacts
- the missing `output` case uses `{}` semantics safely
- `failure_gate` remains `invocation_validation` for structural misuse
- `exc.audit_artifact` remains attached before the exception propagates

## 18. Testing and Replay Requirements

`v0.3.2` must add split-mode coverage without weakening the existing regression
bar.

Required coverage areas:

- Phase A PASS / FAIL matrix
- Phase B PASS / FAIL matrix
- `PreCallResult` immutability and one-time consumption
- unified backward compatibility
- decorator default mode vs split mode
- instance-scoped split methods
- async parity
- pre-pipeline fail artifacts
- schema `v1.3` validation
- golden replays for split and unified parity cases

Tests must not require runtime stale-token detection, because logical staleness
is not a supported runtime check in this design.

## 21. Implementation Risks and Sequencing Summary

The highest-risk implementation work is still concentrated in the enforcement
core.

Recommended sequence:

1. freeze ADR and artifact contract
2. implement schema `v1.3`
3. factor validation and enforcement core into Phase A / Phase B internals
4. add public split exports and instance methods
5. update `@governed`
6. add telemetry split spans
7. run regression and golden replays
8. update docs and demo surface

Primary risks:

- drift between unified and split behavior
- double-emission bugs
- incorrect pre-pipeline FAIL artifact handling
- accidental policy reload in Phase B
- misuse of `PreCallResult` in manual host flows

These risks are manageable if the invariants in this document are treated as
binding implementation constraints.
