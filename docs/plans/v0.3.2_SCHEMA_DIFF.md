# v0.3.2 Schema Diff — `audit_artifact.schema.json` v1.2 -> v1.3

Date: 2026-04-04
Status: Proposed

---

## 1. Summary

Schema v1.3 is an additive evolution of v1.2.

- No existing required field is removed.
- No existing field type is narrowed.
- New split-enforcement metadata fields are optional.
- Existing v1.2 artifacts remain valid under v1.3.

---

## 2. Exact JSON Schema Changes

### 2.1 Version Constant

Current runtime constant:

- `aigc/_internal/audit.py:25` sets `AUDIT_SCHEMA_VERSION = "1.2"`.

Change:

- set to `"1.3"`.

### 2.2 Metadata Field Expansion

Current schema:

- `metadata` is an unconstrained object with `maxProperties: 100` (`schemas/audit_artifact.schema.json:76-79`).

Proposed v1.3 additive metadata properties (descriptive; still optional):

```json
{
  "metadata": {
    "type": "object",
    "maxProperties": 100,
    "properties": {
      "enforcement_mode": {
        "type": "string",
        "enum": ["unified", "split", "split_pre_call_only"],
        "description": "Execution mode used for this artifact"
      },
      "pre_call_gates_evaluated": {
        "type": "array",
        "items": { "type": "string" },
        "maxItems": 1000,
        "description": "Ordered list of Phase A gates completed"
      },
      "post_call_gates_evaluated": {
        "type": "array",
        "items": { "type": "string" },
        "maxItems": 1000,
        "description": "Ordered list of Phase B gates completed"
      },
      "pre_call_timestamp": {
        "type": "integer",
        "minimum": 0,
        "description": "Unix epoch seconds when Phase A completed"
      },
      "post_call_timestamp": {
        "type": "integer",
        "minimum": 0,
        "description": "Unix epoch seconds when Phase B completed"
      }
    },
    "additionalProperties": true
  }
}
```

Notes:

- `additionalProperties: true` on metadata preserves v1.2 flexibility and existing extensions (`risk_scoring`, `custom_gate_metadata`, etc.).

## 3. Failure Gate Enum Review

Current enum (`schemas/audit_artifact.schema.json:50-53`):

- `invocation_validation`
- `role_validation`
- `precondition_validation`
- `schema_validation`
- `postcondition_validation`
- `feature_not_implemented`
- `tool_validation`
- `guard_evaluation`
- `condition_resolution`
- `sink_emission`
- `risk_scoring`
- `custom_gate_violation`

### Decision

No enum additions required for split enforcement.

Rationale:

- missing, invalid, or reused `PreCallResult` inputs can map to `invocation_validation`.
- phase-specific behavior is represented through metadata (`enforcement_mode` and phase gate lists), not new failure-gate taxonomy.

---

## 4. Metadata Structure Changes

v1.2 model:

- single pipeline evidence list via `metadata.gates_evaluated`.

v1.3 model:

- unified mode: continue `metadata.gates_evaluated` and set `metadata.enforcement_mode = "unified"`.
- split mode:
  - `metadata.enforcement_mode = "split"`
  - `metadata.pre_call_gates_evaluated`
  - `metadata.post_call_gates_evaluated`
  - timestamps per phase
- split pre-call fail mode:
  - `metadata.enforcement_mode = "split_pre_call_only"`
  - `metadata.pre_call_gates_evaluated` present
  - `metadata.post_call_gates_evaluated` absent or empty

Compatibility guidance:

- readers should use `enforcement_mode` when present.
- legacy readers can continue to process core artifact fields without mode awareness.

---

## 5. Backward Compatibility Proof

1. New fields are optional and additive.
2. Existing required list is unchanged.
3. Existing enum values are unchanged.
4. Existing top-level field semantics are unchanged.
5. Existing v1.2 artifacts validate under v1.3.
6. Existing consumers that ignore unknown metadata keys remain unaffected.

Therefore v1.3 is backward compatible with v1.2 producers and consumers.
