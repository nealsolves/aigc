# `v0.9.0` PR-07 Beta Proof Plan

Date: 2026-04-17
Status: Proposed
Target branch: `feat/v0.9-07-beta-proof`

---

## Goal

Prove that the default `v0.9.0` adopter journey is real from a clean
environment, not just architecturally planned.

PR-07 is the mandatory stop-ship checkpoint. It must prove that a first adopter
can:

1. install the base package in a fresh environment
2. run `aigc workflow init`
3. reach a first PASS on a local host-owned workflow
4. hit one intentional, understandable failure
5. diagnose that failure with `aigc workflow doctor` or `aigc workflow lint`
6. apply the documented fix and return to PASS

All of that must happen through public docs, public APIs, starter assets,
examples, and the maintained demo only.

## PR-07 Contract

- The default adopter path must succeed without Bedrock or A2A.
- No quickstart, example, starter recipe, demo code, or docs snippet may
  import from `aigc._internal`.
- No mandatory hand-authoring of advanced manifests is allowed on the default
  path.
- PR-07 may prove workflow and evidence visibility through emitted workflow
  artifacts, correlated invocation evidence, and demo UI views, but it must
  not ship the PR-09 `workflow trace` or `workflow export` CLI surfaces early.
- The maintained demo must use real SDK and backend behavior. No canned
  "pretend governance" backend responses.
- README and public contract docs must remain honest about shipped vs unreleased
  surfaces until the version bump and beta-freeze work lands later in the train.

### 3. Proof Scenarios

PR-07 should standardize on three concrete proof paths:

- First PASS path:
  `minimal` starter from clean environment to `COMPLETED`.
- Second PASS path:
  `standard` starter path to prove that at least two public-import-only starter
  examples are green.
- Intentional failure-and-fix path:
  `regulated-high-assurance` starter with an intentionally broken
  source-required/provenance path, followed by diagnosis, documented fix, and a
  rerun back to PASS.

Why this split:

- `minimal` is the fastest first success path.
- `standard` proves the default path is not a one-off toy.
- `regulated-high-assurance` gives a fail-closed, policy-meaningful failure
  that can be explained and fixed without depending on PR-08 engine hardening
  or PR-09 export tooling.

### 4. Clean-Environment Proof Harness

Add a dedicated proof harness, preferably
`scripts/validate_v090_beta_proof.py`, that:

- creates a fresh virtual environment
- installs the repo in editable mode with the existing documented install path
- runs the quickstart path end to end
- measures elapsed quickstart time against the `15` minute budget
- verifies emitted workflow artifact shape and status
- runs the intentional failure path
- runs `aigc workflow doctor` and/or `aigc workflow lint`
- applies the scripted fix
- reruns to PASS
- emits a machine-readable summary suitable for release-gate evidence

The harness should fail closed if:

- the default path requires `_internal` imports
- the quickstart depends on hidden maintainer setup
- the failure path cannot be diagnosed with the public tooling
- the fix path diverges from the public docs

### 5. Demo Slice

Add one narrow `v0.9.0` workflow-governance slice to the maintained demo
instead of retrofitting every existing `v0.3.x` lab.

Required demo flows:

- one `start here` flow that mirrors the workflow quickstart
- one intentional failure-and-fix flow
- one governed-versus-ungoverned comparison
- one workflow/evidence visibility view

Demo implementation rules:

- use the real React + FastAPI demo stack already maintained in-repo
- use real SDK calls on the backend
- use emitted workflow artifacts and correlated invocation evidence for
  visibility
- do not invent a fake trace/export backend before PR-09
- keep the change set narrow: one dedicated page or route, one backend path,
  help copy updates, and focused smoke tests

### 6. Tests and CI

Add or extend coverage for:

- clean-environment quickstart smoke
- quickstart time-budget proof
- failure-and-fix end-to-end proof
- public-import-only validation across:
  docs snippets, examples, starters, and demo code
- demo smoke coverage for the new workflow path
- doc parity checks for:
  first-adopter docs order, required page presence, and honest shipped vs
  unreleased wording

Important repo-specific constraint:

- root `pytest` is configured for `tests/` only, so demo backend verification
  must be run explicitly or mirrored into root test coverage.

Expected verification commands for PR-07:

- `python -m pytest -v`
- `flake8 aigc`
- `python scripts/check_doc_parity.py`
- `pytest demo-app-api/tests -q`
- `npm --prefix demo-app-react test`
- `npm --prefix demo-app-react run build`

## Exit Gate

PR-07 is complete only if all of the following are true:

- a clean environment can complete the quickstart within the `15` minute target
- the docs-to-working-app path reaches a first PASS without internal-code
  reading
- both `minimal` and `standard` starter examples are green and public-import
  only
- the intentional failure-and-fix scenario is green end to end
- `workflow doctor` and/or `workflow lint` explain that failure clearly enough
  for a first adopter to recover
- workflow and evidence visibility are shown through emitted artifacts and demo
  views without pulling PR-09 surfaces forward
- the default path requires neither Bedrock nor A2A
- no advanced manifest authoring is required on the default path
- the maintained demo uses real backend behavior for the new flow
- release-truth docs, parity checks, and verification commands are all green

## Bottom Line

PR-07 should be treated as product proof, not as another feature PR.

If the quickstart, failure-and-fix path, release docs, and maintained demo do
not form one coherent public journey from clean install to PASS, then the
`v0.9.0` beta is not ready, regardless of how much engine or adapter work lands
afterward.
